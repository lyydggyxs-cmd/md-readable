# Markdown → HTML 简报转换器

将 Markdown 线性文本转化为人类可空间化消费的 HTML 视图。核心原则：**不压缩信息，重组信息**。

## 核心理念

AI 产出的线性 Markdown 和人类空间化视觉消费之间存在 6-7 个数量级的编码成本不对称。这个 skill 建造一个中介层——将 Markdown 解析为 IR（中间表示），然后用 v2.0 设计系统渲染为 HTML。

**这不是"摘要"工具。所有内容都保留——只是换了一种组织方式。**

## 工作流总览

```
读取 MD → 解析三层结构 → 按模板组装 HTML → 写入同目录同名 .html
```

---

## Step 1：读取并分析 Markdown

读取整个 Markdown 文件。边读边将内容归入三个层：

| 层 | 定位 | 从 MD 中找什么 |
|---|------|--------------|
| **Layer 1 — 信号层** | 页面顶部，3 秒定向 | 核心结论句、置信度表述、关键数字（≤3 个）、翻转前提（"如果 X 错了，结论会翻转"类表述） |
| **Layer 2 — 推理层** | 页面主体，选择性深入 | H2/H3 章节、论证段落、对比分析、推理步骤、表格 |
| **Layer 3 — 验证层** | 页面底部，默认折叠 | 参考来源、文献引用、局限性、替代路径、注释 |

**关键**：不要跳过任何 Layer 2 的内容。`<details>` 折叠是渐进披露的手段——所有推理内容必须保留在折叠区内。`<summary>` 提供"信息气味"让读者在展开前知道里面有什么。

---

## Step 2：构建 Layer 1 — 信号卡（SCQA 结构）

从 MD 中提取以下内容，填入信号卡模板：

### SCQA 提取规则

| 元素 | 提取来源 | 要求 |
|------|---------|------|
| **S (Situation)** | 开篇背景描述 | 1-2 句，建立共识 |
| **C (Complication)** | 问题陈述、矛盾、变化 | 1-2 句，制造张力 |
| **A (Answer)** | 核心结论 | 完整断言句，≤28 字（中文） |
| **Metrics** | 关键数字/指标 | 1-3 个，每个带数字+标签 |
| **Premises** | "如果…错了"类表述 | ≤3 条，标明翻转后果 |

如果 MD 没有显式的 SCQA 结构，从内容中推断。信号卡是最重要的元素——它给读者 3 秒定位。

### 置信度判断

从 MD 中识别置信度信号：
- **高**：有明确证据/数据支撑、多来源交叉验证、作者明确标注"高置信度"
- **中**：有推理但缺实证、单一来源、作者标注"推断"/"可能"
- **低**：纯推测、缺乏证据、作者标注"不确定"/"需要验证"

---

## Step 3：构建 Layer 2 — 推理块

对每个 MD 中的主要章节（H2/H3）：

### 章节标题
将 H2 转为 section 标题。

### 推理块（每个主张一个块）

1. **断言标题**：改写为完整断言句，不是主题标签（P1：断言优先）
   - ✅ `前提→推理→结论的完整链条`
   - ❌ `关于 X 的分析` 或 `X 的问题`
2. **摘要**：1-2 句，始终可见——给 F-pattern 扫描者"信息气味"（P3：渐进披露）
3. **置信度**：高/中/低，用左边框颜色编码
4. **完整推理**：放在 `<details>` 内，逐步展开
5. **来源**：折叠在推理详情内部

### 关键规则

- **一容器一主张**（P2）：标题里出现"和"→ 拆成两个推理块
- **保留所有内容**——只是换容器，不删内容
- 展开按钮带上"信息气味"：`展开完整推理（3 个步骤 · 预计阅读 2 分钟）`——具体优于模糊

### 嵌套约束
推理步骤嵌套 ≤ 3 层：信号卡 → 推理块 → 推理步骤/源引用

---

## Step 4：构建 Layer 3 — 验证层

收集：
- 参考来源（编号、名称、链接如可用）
- 局限性/边界条件
- 替代路径/方案
- 注释

组织为折叠区，放在页面底部。保留所有引用，不要省略。

---

## Step 5：插入视觉断点（P9：内容节奏）

每 2-3 个推理块后，插入一个视觉断点防止"文本墙"疲劳：

| MD 中出现 | 使用的视觉断点组件 |
|----------|-----------------|
| A vs B 对比分析 | **对比表** `<table class="comparison-table">` |
| 前提→推理→结论的链条 | **推理链可视化** `<div class="inference-chain">` |
| 多个假设/前提列表 | **假设条件卡** `<div class="assumption-card">` |
| 特别有洞见的陈述 | **关键引语** `<blockquote class="insight-quote">` |

**节奏规则**：
- 不要连续 2 个以上推理块等密度且无视觉断点
- 一页不超过 5 个推理块（超过工作记忆容量 → 考虑拆成子报告）
- 第一个推理块最详细，后续可逐步加速

---

## Step 6：组装 HTML 并输出

1. 读取 `references/design-system.md` 获取完整 HTML 骨架和 CSS
2. 按照三层架构填充内容
3. 添加阅读导航（跳转链接到各章节）
4. 添加 footer（Agent 名、日期、原始 MD 链接）

**输出路径**：`<MD 所在目录>/<MD 文件名（不含 .md）>.html`

---

## 设计决策检查表（输出前逐项自查）

对每个视觉决策确认：

| # | 检查项 | 对应原则 |
|---|--------|---------|
| 1 | 每个推理块标题是完整断言句（不是"关于 X"）？ | P1 断言优先 |
| 2 | 每个推理块只包含一个主张？标题里出现"和"→ 拆分 | P2 一容器一单元 |
| 3 | 非关键信息放在折叠区？首屏只展示结论+摘要 | P3 渐进披露 |
| 4 | 相关元素物理靠近，无关元素分离？（不用颜色/边框定义关系） | P4 空间编码 |
| 5 | Squint Test：模糊后最显眼的 3 个元素就是最重要的 3 个？ | P5 三层视觉层次 |
| 6 | 间距够用吗？够用的话——加倍了吗？ | P6 留白是主动元素 |
| 7 | 强调色出现在 ≤ 2 个元素上？ | P7 单色+一强调色 |
| 8 | 每个 CSS 规则都在传递信息？有没有纯装饰？ | P8 信噪比 |
| 9 | 有没有连续 2+ 个推理块等密度？有 → 插入视觉断点 | P9 内容节奏 |
| 10 | 任何间距/颜色/字号可以追溯到系统 token？ | P10 系统性 |
| 11 | 重要信息不在右下角？ | 死区避免 |
| 12 | 展开按钮带信息气味标签？（具体 > 模糊） | App UI 智慧 |
| 13 | 推理步骤不超过 3 层嵌套？ | 认知负荷 |
| 14 | 正文颜色是 #333（不是 #000）？ | 中文排版 |

## 禁止项（信噪比审计）

以下做法在 HTML 简报中**禁止出现**。它们是"装饰"而非"信息"——加到系统里只会增加噪音：

- ❌ 任何 gradient（渐变）
- ❌ border-radius > 12px
- ❌ emoji 作为标题/CTA 装饰（置信度指示器的 🟢🟡🔴 例外）
- ❌ 3 列以上重复卡片布局
- ❌ 加载外部字体或图标库
- ❌ 过度阴影（最大 `0 4px 12px rgba(0,0,0,0.08)`）
- ❌ 纯装饰的边框、分隔线、背景色
- ❌ 不在 spacing scale 中的任意 margin/padding 值
- ❌ 不在 type scale 中的任意 font-size 值
- ❌ 闪烁、脉冲、无限循环动画
- ❌ 彩色背景（假设卡等语义例外除外）
- ❌ 为不同推理类型使用不同颜色编码
- ❌ 重要内容放在右下角（死区）
